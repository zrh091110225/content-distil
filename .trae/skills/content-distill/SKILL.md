# Content Distill

把一批文本语料提炼成一套可运行、可验证、可复用的内容系统，而不只是做一份泛泛总结。

这个 skill 的目标不是“摘几个高频词”，而是同时完成四件事：

1. 提炼内容系统：领域结构、概念口径、信念/偏好、心智模型。
2. 提炼表达系统：文风 DNA、结构模板、禁忌边界、生成约束。
3. 证明它真的成立：holdout 复测、量化指标、人工盲测。
4. 让结果可继续使用：方法文档、Prompt、Rubric、执行版 Style Spec、可选脚本。

## 何时调用

在以下场景调用这个 skill：

- 用户有一批同账号、同作者或同主题语料，想提炼成可复用的方法论或写作规范。
- 用户希望不仅“分析风格”，还要“验证风格”和“复刻风格”。
- 用户想把“像不像”做成可验收流程，而不是凭主观感觉判断。
- 用户要把一次性的提炼过程沉淀成长期可复用的 skill、模板、文档或脚本。

不适合的场景：

- 只有 1 篇短文本，却要求稳定抽出账号级文风。
- 用户只想做摘要、改写、润色，而不需要体系化提炼与验证。
- 没有可追溯语料，只给非常零散的片段，却要求高置信度下结论。

## 默认分析单位与默认原则

- 默认分析单位：`账号级 / 作者级`
- 默认切分方式：`全量训练集 + 20% 分层 holdout`
- 默认验证方式：`量化指标 + 人工盲测`
- 默认交付方式：`文件落盘 + 对话摘要`

硬性原则：

- 不要在信息不足时硬补完，缺口必须显式标注。
- 不要把“文风”只理解成口头禅和表面句式。
- 不要跳过验证直接宣布“已经提炼完成”。
- 不要对用户尚未定义清楚的框架细节自行补设规则。

## 协作式澄清规则

如果用户正在设计或重构提炼框架，必须先澄清再落地，不能擅自补全。

优先确认的决策点：

1. 分析单位
   - 单篇文章
   - 账号级
   - 作者级
   - 多源主题级
2. 验证集策略
   - 按主题分层
   - 按时间分层
   - 不设 holdout
3. 文风框架
   - 是否采用固定层数
   - 每层是否要求典型样本
   - 是否需要生成约束
4. 指标口径
   - A类是否必须程序可计算
   - B类是否允许 LLM 初判 + 人工复核
   - “是否类指标”是否采用 A+B 双轨
5. 评分方式
   - 等权或不等权
   - 是否存在一票否决项

如果边界不清楚，先问再做。不要为了“显得完整”而补设用户没有确认过的分类或指标。

## 推荐交付物

建议输出目录：`outputs/<corpus_name>/`

核心文件：

- `content_index.jsonl`：全量语料索引，含清洗后的正文和基础元数据。
- `holdout_list.md`：验证集清单和抽样逻辑。
- `corpus_summary.json`：训练集统计摘要。
- `taxonomy.md`：领域地图与典型问题。
- `beliefs_models.md`：信念、偏好、心智模型及验证记录。
- `style_dna.md`：账号级风格基线。
- `style_spec.md` 或执行版 style 文档：可直接用于生成任务。
- `validation_report.md`：holdout 复测记录。
- `new_writing.md`：新问题写作样稿。
- `quant_check.md`：量化命中表。
- `blind_test.md`：盲测题面和记录。

配套文件：

- 方法论文档
- Prompt 模板
- Rubric
- Worksheet
- 可选脚本

## 标准工作流

按下面顺序执行，除非用户明确要求跳步。

### 0. 语料归一化

- 统一元数据：标题、日期、来源、正文、链接。
- 清洗无关噪音：导航、票务、导入 frontmatter、装饰性残留。
- 将正文切成可追溯的证据单元，建议以自然段或语义块为最小单位。

### 1. 训练集与 holdout 划分

- 先粗分主题，再做少量人工校正。
- 默认按主题分层抽取约 20% 作为 holdout。
- holdout 不参与框架提炼，只用于复测和验收。

### 2. 从证据到主张

- 从语料中提取可反驳、可推断的主张句。
- 口号句、套话句、纯煽情句降级处理，不要误当模型。
- 每条主张尽量附：证据、成立条件、反例或边界。

### 3. 领域结构与概念口径

- 形成 2-3 层 taxonomy。
- 叶子节点必须能对应“用户实际关心的问题”，而不只是抽象名词。
- 概念解释优先使用语料自己的口径，而不是百科式改写。

### 4. 理念、偏好与取舍机制

- 提炼稳定价值观，而不是逐篇观点拼盘。
- 明确偏好矩阵：在什么条件下偏向什么，不做“一刀切价值判断”。
- 单独记录反对清单和红线。

### 5. 心智模型提炼与降级

- 先多提候选模型，再收敛到 3-7 个高价值模型。
- 每个模型都要写：定义、变量关系、触发条件、推断规则、证据、边界、反例。
- 用三重验证判断它是不是“真模型”：
  - 跨主题复现
  - 有生成力
  - 有排他性
- 通过不足的候选项要降级为启发式，而不是强行保留。

### 6. 文风 DNA 提炼

默认使用 8 层主框架：

1. 标题
2. 开头
3. 结构
4. 句法
5. 词汇
6. 观点
7. 案例
8. 情绪

每层优先记录 7 个字段：

- 层定义
- A类硬量化
- B类半量化
- C类模式识别
- D类禁忌指标
- 典型样本
- 生成约束

### 7. 把分析文档转成执行规范

不要停在“分析版 style_dna”。

继续把它转成可直接用于生成的执行规范：

- 明确写作模式
- 明确模板选择方式
- 明确必命中约束
- 明确禁忌项
- 明确写后自检顺序

如果用户后续要做改写、新问题写作或 A/B 对照，这一步是必须的。

### 8. 新问题写作与闭环验证

至少做一次新问题写作或同主题改写，并完成闭环：

1. 生成文本
2. 解释用了哪些模型和结构模板
3. 做 D 类违例检查
4. 做量化命中检查
5. 做人工盲测或盲评表
6. 记录主要偏差层和下一轮修订方向

## 文风 DNA 的推荐判定口径

如果用户没有另行指定，使用以下默认口径：

### A类：硬量化

- 只纳入程序规则可重复判定的指标。
- 可以使用规则、词表、正则、结构识别。
- 结果必须可复查、可批量跑。

适合放在 A 类的典型内容：

- 标题长度
- 平均句长
- 段落长度
- 问号数量
- 转折词频
- 第一/第二人称频次
- 标题句式分布
- 开头信号出现位置
- 案例段落数

### B类：半量化

- 用于判断强度、明显度、压迫感、口语化程度、反常识程度等。
- 建议由 LLM 初判，再人工复核。
- 默认使用 `1-5` 分。

### C类：模式识别

- 用来总结稳定写法，而不是简单做词频罗列。
- 优先提炼：
  - 常用标题结构
  - 常用开头方式
  - 常用结构模板
  - 常用问题重定义路径
  - 常用案例功能
  - 常用结尾方式

### D类：禁忌指标

- 这是生成质量提升最快的一层。
- 记录“这个账号绝不会怎么写”。
- D 类违例不能被高分 B 类掩盖。

典型 D 类内容包括：

- 禁用词
- 禁用标题方式
- 禁用开头方式
- 禁用情绪动作
- 禁用结构套路
- 禁用悬浮案例

### A+B 双轨

如果某指标本质上是“是否出现”，但仍然有明显度差异，采用 A+B 双轨：

- A：记录有无或占比
- B：记录强弱或显著度

例如：

- 是否前置结论
- 是否制造冲突
- 是否商业转化
- 是否使用对象词或场景词

## 提炼过程中的反模式

出现下面情况时，要主动纠偏：

- 只做高频词表，不做概念口径与边界。
- 只总结“风格像什么”，不写禁忌项和生成约束。
- 只靠训练集自证，不做 holdout 复测。
- 把所有候选模型都留下，导致模型泛滥。
- 用表面句式替代观点结构。
- 用主观形容词替代证据，例如“比较温暖”“比较犀利”，却没有样本和规则。
- 为了量化而量化，把明显依赖理解的指标强行塞进 A 类。

## 输出质量门槛

完成提炼前，至少检查以下问题：

1. taxonomy 是否能回链到具体问题，而不是抽象大词。
2. beliefs/models 是否写清边界和反例。
3. style_dna 是否不是只剩“词汇 + 句式”。
4. 每层是否有典型样本。
5. 是否完成至少一次 holdout 复测。
6. 是否完成至少一次新问题写作或改写验证。
7. 是否有量化命中表。
8. 是否有人工盲测或等价人工核验。

## 对外汇报建议

给用户汇报时，优先按下面顺序组织：

1. 先讲提炼目标和分析单位。
2. 再讲输出物和验证方式。
3. 再给核心发现：
   - 领域结构
   - 核心信念
   - 心智模型
   - 文风 DNA
4. 最后讲验证结果和剩余短板。

如果用户问“还有没有没完成的任务”，要区分：

- 显式任务是否已完成
- 验证闭环是否已完成
- 是否还存在建议性的下一轮优化点

## 最小执行示例

当用户说：

- “帮我从这个账号文章里提炼方法论和文风”
- “我想验证 AI 写得像不像这个号”
- “把这次提炼过程沉淀成一个可复用 skill”

默认动作应是：

1. 确认分析单位、语料范围、holdout 策略、输出物。
2. 先搭方法和验证框架，不直接仓促下结论。
3. 从证据、主张、taxonomy、beliefs/models、style_dna 逐步推进。
4. 在框架明确后，再扩到全量语料。
5. 最后用新问题写作或改写做闭环验证，并沉淀执行版规范。
